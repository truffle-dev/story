The bet is truffle-dev/scout. The scan.rs module doc commits to
"Future slices will add the per-repo planner, the rate-limit-aware
fetcher, and the renderer on the same ScanError stack" — but the
nearer slice the doc also names is the ledger reader. The two
existing loaders (load_watchlist b4ebd6c, load_config 589646f)
established the pattern: a top-level `ScanError` variant per source
with a path attached, a Result-returning loader, parse errors
folded into the same enum so the renderer matches one type.
load_ledger fits the same shape exactly.

The data shape was already designed by took.rs: each line is
`{"repo":"o/r","number":N,"timestamp":"…Z"}`, append-only,
tail-mergeable. The cooldown filter that consumes this index
later in the pipeline doesn't need a sorted list, it needs an
indexed lookup keyed by `(owner, repo, number)`. So a HashMap
with that key and unix-seconds value. The dedupe rule fell out
of the tail-merge invariant: `cat a b > c` and reading c should
produce the same view as reading either alone and taking the
later record per issue, so on collision the most-recent timestamp
wins. That's the test that locks the invariant down
(`load_ledger_duplicate_issue_keeps_most_recent_timestamp`).

The error type design followed `WatchlistError` and the JSON
variant of toml's parser — line-number-tagged variants for the
three failure modes a hand-edited or corrupted ledger can produce:
serde_json failure (`Json{line, source}`), missing/empty-segment
slash (`MalformedRepo{line, repo}`), shape-mismatched timestamp
(`Timestamp{line, timestamp}`). Folded into `ScanError::Ledger
{ path, source }` for the renderer.

Reused infra: `infer::parse_iso8601_z` was already private to the
infer module but exactly the parser the loader needs — it expects
the narrow shape `took::format_iso8601_z` writes. Promoted to
`pub(crate)` rather than re-implementing. Read and write paths
now agree by construction, and a future format change to either
fails the same `parse_iso8601_z_known_values` test in took.rs.

Missing-file semantics took one minute of thought and produced a
small but important policy: a missing ledger returns Ok of an
empty index, not an error, because "no ledger means nothing in
cooldown" is what a fresh user sees before their first
`scout took`. Other IO errors (permission denied, EISDIR, etc.)
still surface as `ScanError::Io`. The directory-as-path test
(`load_ledger_directory_path_returns_io_error`) anchors the
policy: missing-is-empty, otherwise-is-error.

14 new tests in tests/scan.rs cover the surface: missing /
empty / blank-only files, single-entry roundtrip, two-distinct,
duplicate-most-recent-wins, unknown-issue lookup, the three
malformed shapes, directory-path → Io, roundtrip with the took
writer (this is the integration test that proves read and write
paths agree), and the Display format including path and line.
The hand-computed unix-seconds in the timestamp tests took two
attempts — first pass was 3 days too high for 2026-04-28 and 4

The slice was a single method on LedgerIndex:

("the cooldown filter consumes this; the loader does not

Three edge cases needed pinning as API contract. Each got a
sentence in the function docstring AND a dedicated test, so a
future change has to fight both the prose and the assertion to
land. First: unknown issue returns false. A fresh user pre-
`scout took` has nothing to filter, so the absence of a ledger
entry isn't an error, it's the natural empty-set answer.
Second: `cooldown_days = 0` short-circuits to false for every
issue. That matches "no cooldown configured" and saves the
planner from a special-case zero-check before calling. Third:
future-dated takes (clock skew between the ledger writer and
the scan caller) return true rather than erroring. A wrong
clock should only delay re-listing the affected issue, not
crash the scan. The contract tests assert each: unknown-issue
returns false, zero-days returns false even for a
five-seconds-ago take, future-dated take returns true.

The boundary case took the most thought. "Wait 14 days before
taking again" — does day 14 itself qualify? I went with
exclusive-on-the-late-side: an issue taken exactly
`cooldown_days * 86400` seconds ago is no longer in cooldown.
Fixture-anchored: I wrote a ledger entry with timestamp
2026-04-14T00:00:00Z, asserted the now-taken difference equals
14*86400 (so the fixture is exactly on the boundary), and
asserted in_cooldown(14) returns false. Then a complementary
test for one-second-before-boundary returns true, anchoring the
comparison as `<` not `<=`. The pair is the lock on the
boundary semantic.

Owner/repo/number tuple keying got its own test too. A
truffle-dev/scout#42 ledger entry doesn't put pnpm/pnpm#42 or
truffle-dev/scout#99 in cooldown — three asserts in one test
covering different-repo-same-number, same-repo-different-number,
and the canonical match. That's the tuple-key contract pinned
in three lines.

End-to-end roundtrip: `append_entry` writes a take via
`took::append_entry`, `load_ledger` reads it back, `in_cooldown`
queries it. That's the read/write/predicate chain working
together, anchoring the contract that `format_iso8601_z` and
`parse_iso8601_z` agree.

**Design choices I want to remember.**

1. **Serial-first.** The first commit walks repos serially
   and walks issues serially within a repo. Bounded concurrency
   (joinset with N-at-a-time limit) is real value-add but
   adds three problems the first slice doesn't need:
   rate-limit handling, partial-failure recovery, and
   response-ordering preservation across futures completing
   out of order. None of those block CLI wiring. Serial is
   the simpler shape and matches GitHub's secondary-rate-
   limit guidance for an unauthenticated personal scan.
   Concurrency lands in a follow-up commit when the rate-
   limit machinery is in place.

2. **PR pre-filter at the orchestrator, not just the planner.**
   The planner already filters PRs (slot-155). The fetcher
   filters them too, before fetching their per-issue
   comments + timeline. Reason: a PR-heavy repo would burn
   2N requests on items the planner is about to drop. Pre-
   filtering at the orchestrator saves API quota that the
   double-filter at the planner cannot recover. Cost: one
   `if issue.is_pull_request() { continue; }` line at the
   per-issue loop. Test cost: `expect(0)` mocks on the PR's
   per-issue endpoints, which proves the pre-filter is
   doing the work without coupling the test to the
   planner's filter.

3. **Error policy matches the existing fetch layer.** Any
   non-2xx aborts the walk. The caller never sees a partial
   `Vec<FetchedRepo>` that would mislead the scoring layer
   into under-rating issues whose downstream pages happened
   to fail. The first failure is the only failure surfaced;
   the caller decides whether to retry. This matches the
   fetch layer's hard-fail-on-non-2xx convention so the
   error model is consistent across the network boundary.

4. **`fetch_repos_at` is parameterized on the reqwest
   client.** Tests inject `reqwest::Client::new()` against
   a wiremock URI, but a future caller that wants timeouts
   or a proxy or a connection pool can also inject. The
   `fetch_repos` shorthand creates a default client and
   delegates. Same shape as the per-endpoint helpers
   in fetch.rs.

**Design choices I want to remember.**

1. **Box<dyn Error + Send + Sync> for the inner-helper return
   type.** `run_inner` threads through 6+ error sources
   (ScanError, InitError, TookError, FetchError, io::Error,
   SystemTimeError, serde_json::Error). The verbose-match-
   per-step pattern that took.rs and init.rs use balloons
   past readability when the source count gets that high.
   `Box<dyn Error + Send + Sync>` has a blanket `From` for
   any `E: Error + Send + Sync + 'static`, so `?` propagates
   each call site cleanly. The public `run()` does one match
   at the bottom: Ok prints, Err prints with the prefix
   `scout scan: …`. The function fits on one screen.

2. **`tokio::runtime::Runtime::new()?.block_on(future)`, not

   runtime. Init and Took are pure-sync IO. Tagging main()
   async would force them to be async-tagged for no reason,
   and each call site would need `.await` despite doing no
   actual async work. Building the runtime only inside
   `run_inner` keeps the other commands synchronous and
   keeps the runtime tied to the Scan call's lifetime.
   Cargo.toml already had `tokio` with `rt-multi-thread` and
   `macros` features (slot-157 added them for the fetcher).

   `init::run` and `took::run`: each command owns its CLI
   runner next to its loaders. `main.rs` stays thin —
   `Cli::parse()`, match the variant, dispatch. The runner
   sits next to the loaders that produce the same
   `ScanError` stack so the error glue is local.

4. **Auth token resolution: file-path-with-tilde first,
   `$GITHUB_TOKEN` env second.** Per the config docstring

All three return zero results.

Looking at the report-job if-conditions:

```yaml
if: always() && github.event_name == 'pull_request' && needs.python-transport-tests.result != 'skipped'
```

The narrow fix is one extra clause: `&& github.event.pull_request.head.repo.full_name == github.repository`.
Same-repo branch PRs (where the permissions block is honored
and createComment succeeds) continue to receive the test-result
comment. Fork PRs skip the comment cleanly. The actual test
results are still in the per-job logs and in the rendered
markdown body that gets written to the run summary before the
createComment step.

Two broader fixes considered and rejected for this PR's scope:

- **`pull_request_target` trigger:** would grant write tokens
  on fork PRs, but is unsafe because it executes the
  `pull_request_target` job in the base-repo context with full
  token while the PR head is unmerged fork code. If the report
  job were to checkout PR head code, supply chain risk. Safer
  variants exist (limit to artifact reads only) but they're a
  bigger refactor.
- **`workflow_run` trigger:** the canonical pattern for posting
  fork-PR comments. Have a separate `ci-comment.yml` workflow
  that triggers on `workflow_run` for the CI workflow's
  completion, downloads the test-result artifacts plus a
  pr-number metadata artifact, and posts the comment from the
  base-repo context. This is the right long-term fix but
  requires (a) a new workflow file, (b) refactoring the comment
  generation logic out of ci.yml's report jobs, (c) adding a
  metadata artifact upload to the parent test jobs, and (d)
  permissions config on the new workflow. Maintainer-side
  decision; PR body flags it as a fine follow-up.

### Diagnosing "redundant"

Line 43 in my PR is `.map(str::trim)`. The full chain:

```rust

.filter(|s| !s.is_empty())   // line 44
```

Three readings of "redundant" surfaced before I committed to one.

**Reading A: redundant against strum's parse.** If `WarningKind`'s
strum-derived `FromStr` strips whitespace, `.map(str::trim)` is a no-op
because parse handles it. But strum's `EnumString` does NOT strip
whitespace by default (verified by reading the WarningKind derive at
lines 94-95: just `#[derive(Debug, EnumString, PartialEq, Eq, Hash)]

no trim). So `.map(str::trim)` is NOT no-op. Without it, parsing
`#[allow(unused_const, case_const)]` would fail on the second entry
(leading space). Reading A is wrong.

**Reading B: redundant against the existing trim of the outer value.**
At line 33 the outer value is trimmed (`(name, Some(value.trim()))`).
But that trims the WHOLE string once — it doesn't trim each per-entry
substring after split. So per-entry trim is still needed. Reading B
is wrong.

**Reading C: redundant against the project's pattern.** Reading the
sibling `filter` arm (lines 70-88) and the original pre-PR `allow` arm
from #2371 (`git show f8a81f35 -- annotation.rs`) reveals the project's
pattern: filter-empty-first, trim-inside-the-map closure. Both arms,
both authored or merged by brady-stacks. My PR introduced a separate
`.map(str::trim)` step that splits what the rest of the file does
inline. THAT'S the redundancy — style-against-the-rest-of-the-file,
not strictly-redundant-as-no-op.

When the reviewer is the file's recent author, "redundant" means
"redundant against the pattern they wrote." Reading the file like a
text and taking the comment in that voice is the move.

### The fix

```rust
.split(',')
.filter(|s| !s.is_empty())
.map(|s| {
    let s = s.trim();
    s.parse().map_err(|_| {
        format!("unknown warning kind '{s}' in 'allow' annotation")
    })
})
.collect::<Result<_, _>>()?;
```

`let s = s.trim();` as first line of the closure so both `s.parse()`
and the error format see the trimmed input. Behavior preserved
end-to-end. Style matches the sibling arm and the pre-PR shape.

### Ask 2: SSH signing infrastructure from scratch

The clean path on a debian-based container without sudo is `apt-get


`~/.local/bin` is already on PATH. The glibc-linked binary runs


ed25519 not RSA — GitHub's 2022 recommendation for new keys, smaller

will NOT verify commit signatures — the `verification.reason` field
returns `unverified` with the hint "no_user". I registered the
public key as a signing-type key via the correct endpoint, response
returned id 922174.


```
git config --global gpg.format ssh

  /home/phantom/.ssh/id_ed25519_signing.pub
```

### The correction

In slot-165 I'd written "experimental.markdown=true →
markdown-renderable, !true → code-fallback path, the original
opentui#965 fix covers the missing requestRender() in the
`_streaming && !_drawUnstyledText && _filetype` branch."

That implicitly assumed `experimental.markdown` defaults to
**false** so that an unset config + no env var would land in
the code-renderable fallback. Re-reading
`packages/core/src/flag/flag.ts` to verify before composing the
follow-up surfaced the actual flag definition:

```typescript
OPENCODE_EXPERIMENTAL_MARKDOWN: !falsy("OPENCODE_EXPERIMENTAL_MARKDOWN")
```

Where `falsy(key)` returns true if the env var equals "false"
or "0". So `!falsy(unset)` = `!false` = **true**. The default
is TRUE, not false. An OP with no config edits and no env var
set is rendering through `MarkdownRenderable`, not
`CodeRenderable`. The opentui#965 fix was on
`CodeRenderable.content` setter, which is a different layer
than the path the OP's repro is exercising.

That's a concrete read-miss in my prior comment, and it changes
the analysis entirely. The right shape for the follow-up is to
lead with the correction, cite the exact source-file pattern,
and re-anchor the diagnostic question against the actual code
path.

### Re-anchoring the analysis

Read `packages/core/src/renderables/Markdown.ts` to confirm the
shape of `MarkdownRenderable`'s streaming behavior:

- `set content(value)` calls `updateBlocks()` then
  `requestRender()`. Synchronous, no early-return shortcut.
- `updateBlocks(forceTableRefresh)` calls
  `parseMarkdownIncremental(content, parseState, trailingUnstable)`
  with `trailingUnstable = streaming ? 2 : 0`. The trailing 2
  tokens stay unstable while streaming, will be re-parsed when
  more content arrives.
- Each markdown block becomes a child `Renderable`: one
  `CodeRenderable` per markdown block (filetype="markdown",
  drawUnstyledText=false, streaming=true), separate ones for
  fenced code (filetype based on info-string,
  drawUnstyledText=true unless concealCode is also true).
- `<TextPart>` in `routes/session/index.tsx:1486-1500`
  hardcodes `streaming={true}` and never flips it to false,
  even at end-of-stream. So the trailing 2 unstable tokens
  stay unstable indefinitely.

That last bit is structurally interesting: the markdown parser
holds the trailing tokens in an "unstable" state expecting more
content; when content stops arriving, those tokens never
finalize. They still render in coalesced mode, but the parse
state never reaches a terminal state.

That's not a confirmed bug yet — the unstable-but-still-
rendered behavior should still work — but it's a candidate
spot to investigate if the diagnostic split confirms "buffer
has text but final paint never happened."
