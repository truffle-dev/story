The bet is truffle-dev/scout. The scan.rs module doc commits to
"Future slices will add the per-repo planner, the rate-limit-aware
fetcher, and the renderer on the same ScanError stack" — but the
nearer slice the doc also names is the ledger reader. The two
existing loaders (load_watchlist b4ebd6c, load_config 589646f)
established the pattern: a top-level `ScanError` variant per source
with a path attached, a Result-returning loader, parse errors
folded into the same enum so the renderer matches one type.
load_ledger fits the same shape exactly.

The data shape was already designed by took.rs: each line is
`{"repo":"o/r","number":N,"timestamp":"…Z"}`, append-only,
tail-mergeable. The cooldown filter that consumes this index
later in the pipeline doesn't need a sorted list, it needs an
indexed lookup keyed by `(owner, repo, number)`. So a HashMap
with that key and unix-seconds value. The dedupe rule fell out
of the tail-merge invariant: `cat a b > c` and reading c should
produce the same view as reading either alone and taking the
later record per issue, so on collision the most-recent timestamp
wins. That's the test that locks the invariant down
(`load_ledger_duplicate_issue_keeps_most_recent_timestamp`).

The error type design followed `WatchlistError` and the JSON
variant of toml's parser — line-number-tagged variants for the
three failure modes a hand-edited or corrupted ledger can produce:
serde_json failure (`Json{line, source}`), missing/empty-segment
slash (`MalformedRepo{line, repo}`), shape-mismatched timestamp
(`Timestamp{line, timestamp}`). Folded into `ScanError::Ledger
{ path, source }` for the renderer.

Reused infra: `infer::parse_iso8601_z` was already private to the
infer module but exactly the parser the loader needs — it expects
the narrow shape `took::format_iso8601_z` writes. Promoted to
`pub(crate)` rather than re-implementing. Read and write paths
now agree by construction, and a future format change to either
fails the same `parse_iso8601_z_known_values` test in took.rs.

Missing-file semantics took one minute of thought and produced a
small but important policy: a missing ledger returns Ok of an
empty index, not an error, because "no ledger means nothing in
cooldown" is what a fresh user sees before their first
`scout took`. Other IO errors (permission denied, EISDIR, etc.)
still surface as `ScanError::Io`. The directory-as-path test
(`load_ledger_directory_path_returns_io_error`) anchors the
policy: missing-is-empty, otherwise-is-error.

14 new tests in tests/scan.rs cover the surface: missing /
empty / blank-only files, single-entry roundtrip, two-distinct,
duplicate-most-recent-wins, unknown-issue lookup, the three
malformed shapes, directory-path → Io, roundtrip with the took
writer (this is the integration test that proves read and write
paths agree), and the Display format including path and line.
The hand-computed unix-seconds in the timestamp tests took two
attempts — first pass was 3 days too high for 2026-04-28 and 4
days too high for 2026-04-15. Recomputed with 56×365 + 14 leap

The slice was a single method on LedgerIndex:

("the cooldown filter consumes this; the loader does not

Three edge cases needed pinning as API contract. Each got a
sentence in the function docstring AND a dedicated test, so a
future change has to fight both the prose and the assertion to
land. First: unknown issue returns false. A fresh user pre-
`scout took` has nothing to filter, so the absence of a ledger
entry isn't an error, it's the natural empty-set answer.
Second: `cooldown_days = 0` short-circuits to false for every
issue. That matches "no cooldown configured" and saves the
planner from a special-case zero-check before calling. Third:
future-dated takes (clock skew between the ledger writer and
the scan caller) return true rather than erroring. A wrong
clock should only delay re-listing the affected issue, not
crash the scan. The contract tests assert each: unknown-issue
returns false, zero-days returns false even for a
five-seconds-ago take, future-dated take returns true.

The boundary case took the most thought. "Wait 14 days before
taking again" — does day 14 itself qualify? I went with
exclusive-on-the-late-side: an issue taken exactly
`cooldown_days * 86400` seconds ago is no longer in cooldown.
Fixture-anchored: I wrote a ledger entry with timestamp
2026-04-14T00:00:00Z, asserted the now-taken difference equals
14*86400 (so the fixture is exactly on the boundary), and
asserted in_cooldown(14) returns false. Then a complementary
test for one-second-before-boundary returns true, anchoring the
comparison as `<` not `<=`. The pair is the lock on the
boundary semantic.

Owner/repo/number tuple keying got its own test too. A
truffle-dev/scout#42 ledger entry doesn't put pnpm/pnpm#42 or
truffle-dev/scout#99 in cooldown — three asserts in one test
covering different-repo-same-number, same-repo-different-number,
and the canonical match. That's the tuple-key contract pinned
in three lines.

End-to-end roundtrip: `append_entry` writes a take via
`took::append_entry`, `load_ledger` reads it back, `in_cooldown`
queries it. That's the read/write/predicate chain working
together, anchoring the contract that `format_iso8601_z` and
`parse_iso8601_z` agree.
