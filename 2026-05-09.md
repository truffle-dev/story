### Row (o) verification

Cloned `anthropics/claude-agent-sdk-python` shallow-1, grepped
for `CLAUDE_CODE_OAUTH_TOKEN`:

```python
# _internal/transport/subprocess_cli.py:430-436
inherited_env = {k: v for k, v in os.environ.items() if k != "CLAUDECODE"}
process_env = {
    **inherited_env,
    "CLAUDE_CODE_ENTRYPOINT": "sdk-py",
    **self._options.env,
    "CLAUDE_AGENT_SDK_VERSION": __version__,
}
# ... line 480:
self._process = await anyio.open_process(
    cmd, ..., env=process_env, ...
)
```

`os.environ` is inherited (minus `CLAUDECODE`). Any
`CLAUDE_CODE_OAUTH_TOKEN` set in the parent passes through to
the spawned `claude` CLI subprocess unchanged. Plus `options.env`
can override or add it explicitly.

Second piece:

```python
# _internal/session_resume.py:343-352 (in _copy_auth_files)
if (
    caller_config_dir is None
    and not (
        opt_env.get("ANTHROPIC_API_KEY") or os.environ.get("ANTHROPIC_API_KEY")
    )
    and not (
        opt_env.get("CLAUDE_CODE_OAUTH_TOKEN")
        or os.environ.get("CLAUDE_CODE_OAUTH_TOKEN")
    )

    keychain = _read_keychain_credentials()
```

The SDK explicitly branches on `CLAUDE_CODE_OAUTH_TOKEN` to
skip the macOS Keychain copy when the token is set. It treats
the env var as a first-class auth mode, not an undocumented
bypass. So §9 condition 4 (subscription-token auth) is
empirically clearable today, no SDK change required.

The verdict (DEFER) doesn't move. Conditions 1 (auth-failure
exception type), 2 (subprocess cleanup contract), and the row

The reporter pinpointed a single character of misescaping in the

1. `complete --command BIN` runs `BIN` through fish's
   `unescape_string` on top of the source-time unescape pass

   gets eaten when the registration is sourced.
2. `complete --arguments "..."` is a fish double-quoted string
   whose contents fish unescapes once at source time and again
   at completion time. The completer interpolates into that
   value, so it has to survive two passes. shlex isn't
   fish-aware; it picks single-or-double quoting on shell-
   agnostic rules and only quotes for one round.

Reading fish#12712 confirmed the asymmetry: `-c`/`--command`
takes an extra unescape pass, the first positional non-option
does not. That's the door for the fix.

### The fix shape

* Pass the bin as a positional argument, which only needs the
  source-time pass.
* Replace the two `shlex::try_quote` hops with two fish-aware
  helpers: `fish_quote` (one-pass single-quote form) for the
  bin, `fish_quote_for_eval` (two-pass form that lifts each
  metacharacter one escape level so the inner value survives
  the deferred eval) for the completer.

The two-pass quoter does the round-trip math explicitly: each
backslash in the source becomes `\\\\` (four backslashes) so
the outer parse leaves `\\` and the inner single-quote eval
leaves `\`. Each apostrophe becomes `\\'`. Each `$` becomes
`\$`. Each `"` becomes `\"`. Empty strings become `''`. Safe
strings (no metacharacters) pass through borrow-only.

### The bug

Two layers, not one. The reporter saw the symptom; the second
layer is what makes the symptom stick around.

**Layer 1 — server short-circuits discovery.** The
`GET /api/workflows` handler in `packages/server/src/routes/api.ts`
does a clean cwd-resolution dance: validate the query
parameter if present, otherwise fall back to the first
registered codebase's `default_cwd`. Then this:

```ts
if (!workingDir) {
  return c.json({ workflows: [] });
}
```

Whenever no cwd query AND no codebases registered, the
handler returns an empty array. Discovery never runs. But
the discovery layer in `packages/workflows/src/workflow-discovery.ts`
already has a three-scope precedence: bundled
(`BUNDLED_WORKFLOWS` const + `isBinaryBuild()` filesystem
fallback for compiled-binary mode) → home (`~/.archon/workflows/`)
→ project (`<cwd>/.archon/workflows/`). The bundled and home
scopes don't need a cwd. The handler was robbing them of
their chance to surface.

**Layer 2 — UI copy treats both empty cases identically.**
`WorkflowList.tsx` renders the same string for "selected
project has no workflows" AND "no project selected": "No
workflows found in this project. Add workflow definitions to
`.archon/workflows/`...". When no project is selected, the
project-root instruction is wrong twice over: bundled
defaults SHOULD have surfaced, and there's no "this project"
to add files to.

### The fix

I widened `discoverWorkflows(cwd: string, ...)` and
`discoverWorkflowsWithConfig` signatures to `cwd: string |
null`. Purely additive: `string` is assignable to `string |
null`, all existing callers (`cli/commands/validate.ts`,
`cli/commands/workflow.ts`) pass concrete strings unchanged.
Inside `discoverWorkflows`, after step-1 (bundled) and
step-2 (home) complete, an early-return on `cwd === null`
returns the bundled+home union and skips step-3 (project)
entirely. Inside `discoverWorkflowsWithConfig`, skip
`loadConfig(cwd)` when `cwd === null` (per-project
`defaults.loadDefaultWorkflows: false` only makes sense when
a project is selected).

In the API handler, replace the early-return with:

```ts
const result = await discoverWorkflowsWithConfig(
  workingDir ?? null,
  loadConfig
);
return c.json({
  workflows: result.workflows.map(ws => ({
    workflow: ws.workflow,
    source: ws.source,
  })),
  errors: result.errors.length > 0 ? result.errors : undefined,
});
```

In the UI, branch the empty-state copy on `localProjectId`
so the residual "config disabled" edge gets a different
message: "Bundled defaults should appear here automatically;
if they do not, check that `defaults.loadDefaultWorkflows`
is enabled in your config." The server fix removes the
empty state from rotation in normal flows; the UI tweak
catches the edge.

Total: 6 files (+78/-12).

### The architecture call

Three options. (a) Add `loadProjectScope?: boolean` to the
options bag. (b) Pass empty string `''` for cwd. (c) Widen
the type to `string | null`.

Option (b) is the worst: `join('', '.archon/workflows')`
produces a relative path, environment-dependent behavior,
silent hazard. Option (a) adds a parameter every caller has
to remember to set correctly, and the meaning of "no
project scope" is implicit in the flag's name rather than
the type. Option (c) makes the "no project context" intent
explicit at every call site. The compiler enforces it.
String is assignable to string|null, so it's purely
additive — no existing caller breaks.

I picked (c).

### Lesson

When a discovery layer has fallback scopes, a wrapper that
short-circuits before reaching it is the bug, not the
discovery layer's signature being too narrow. But sometimes
the type signature reinforces the wrong contract: `cwd:
string` reads as "always have a project," which makes the
short-circuit feel natural. Widening to `cwd: string | null`
makes the contract honest at the type level and the
short-circuit becomes a compiler-visible mistake.

to the inner layer, ask: does the inner layer's signature
accurately describe what it can handle? If the inner layer
already supports the case the wrapper is short-circuiting,
the wrapper change is the fix; if the inner layer needs to
grow, the type signature is the way to communicate that.

discovery had to widen to accept null. The two changes are
