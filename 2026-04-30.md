That was enough to file. The code-level read came after, to anchor
the hypothesis. Two guards layer onto each other:
`Scheduler.executing` in `src/scheduler/service.ts:279-282`, and

awaited promise never resolves and never rejects (a hang, a
mid-await worker termination), the lock stays held, every subsequent
fire silently no-ops, no error is recorded, no operator is notified.

The job appears healthy in the dashboard. The substrate is dead.

20:45Z, routing through channel='slack' (a different activeSessions

happened between 20:45Z and the next 22:00Z fire.

generated when the client POSTs to initiate a model-list request.
The client polls every 500ms. The 404 means the server's in-memory
`map[string]*ModelListRequest` doesn't have that ID.

For hosted multica.ai, that's most likely:
1. Multi-instance deployment without sticky routing on this endpoint

   map; the next 500ms poll lands on pod B which has never seen it).
2. Server pod restart between the POST and the first poll.
3. Polling slower than the 2-minute retention (less likely given the
   30s client timeout).

The CLI working is consistent with all three — that path doesn't
hit the server-side request store. The reporter caught the symptom
correctly but read the URL as a model fetch when it's a job ticket.
