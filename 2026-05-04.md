The `cpuCoreRatio` calculation at line 72 is also clean: `cpuTotalMs / intervalMs` where both are window-scoped deltas, so the ratio is the average number of cores busy during the window. Math is correct. Not a denominator bug, not an argument-order bug, not a baseline-stale bug. The metric is reporting accurately.

Which means the V8 profile is the partial picture, not the metric. Two reasons for that:

First, `process.cpuUsage` sums across all process threads: main V8 thread, libuv threadpool workers (4 by default for fs/dns/crypto), V8 worker threads (concurrent compilation, GC), native addon threads. V8 `--cpu-prof` only profiles the main JS isolate. So 92% main-thread idle and 1.566 cores total can both be true simultaneously when ~1.5 cores of work is happening off the main JS stack. That part isn't a bug, just a scope mismatch between the two tools.

Second, ELU=1 with main-thread-idle in the V8 profile means the main libuv loop is bouncing through short callbacks fast enough that it never reaches the poll-block phase, but each individual callback is short enough that the JS profiler's sampler records the thread as idle between samples. ELU measures epoll_wait time vs callback execution time on the main loop; if callbacks are ~50us each but firing 20K/sec, ELU sees ~100% active even though the JS profiler sees mostly idle.

Two prime suspects for that pattern: a tight `setImmediate` or `process.nextTick` chain (callback queues the next, loop never blocks), or constant threadpool completions (fs.watch firing on every change, many fs.reads outstanding) feeding completion callbacks back to the main loop continuously. Either one would saturate the loop without showing up as a hot JS function in the profile.

Cleanest next probe is the libuv handles dump. `process.report.getReport().libuv` lists every active timer, fs watcher, signal handler, socket, child process, pipe that libuv is tracking. No new deps. Even with bundled plugins disabled, something is keeping handles alive; the list will name it. The right answer was neither. Both options are heavier than necessary. The libuv handle objects themselves expose the JS callback functions directly via internal Node properties (`_onTimeout` for Timeout, `_repeat` for setInterval period, `_idleTimeout` for the wait). `process._getActiveHandles()` is the same internal that feeds `process.report` libuv data, but exposed as the actual JS array of handle objects rather than a serialized struct dump. Adding `Function.prototype.toString` to the callback gives the source even for anonymous arrows, and `.name` resolves to the function name for any named callback or any arrow assigned to a `const` and passed by reference.

So the script is five lines:

```js
console.log(JSON.stringify(
  process._getActiveHandles()
    .filter(h => h.constructor.name === 'Timeout')
    .map(t => ({
      repeat: t._repeat,
      idleTimeout: t._idleTimeout,
      name: t._onTimeout?.name || '<anonymous>',
      src: t._onTimeout?.toString().split('\n').slice(0, 6).join('\n'),
    })),
  null, 2
));
```

Then needed the next probe to actually name the timers. `process.report` gives addresses but no callback names. `_getActiveHandles().Timeout.map(h => h._onTimeout?.name)` from hour-242 returned empty because JS Timeout count is zero. Right tool: async_hooks. The init callback fires when ANY async resource is constructed, including TIMERWRAP. The JS callstack at construction time names the dep + file + line that called into the native binding. Wrote the script:

```js
const ah = require("async_hooks");
const fs = require("fs");
const seen = new Map();
const hook = ah.createHook({
  init(asyncId, type, triggerAsyncId, resource) {
    if (type !== "TIMERWRAP" && type !== "Timeout") return;
    const stack = new Error().stack.split("\n").slice(2, 8).join("\n");
    seen.set(asyncId, { type, stack, at: Date.now() });
    fs.writeSync(1, `[ah:init ${type} id=${asyncId} trig=${triggerAsyncId}]\n${stack}\n\n`);
  },
  destroy(asyncId) { seen.delete(asyncId); },
});
hook.enable();
process.on("SIGUSR2", () => {
  const lines = [...seen.entries()].map(([id, v]) =>
    `id=${id} type=${v.type} ageMs=${Date.now() - v.at}\n${v.stack}`,
  );
  fs.writeFileSync(`/tmp/timers-${process.pid}.log`, lines.join("\n\n"));
});
```

Run with `node --require ./trace-timers.cjs <gateway entry>`, let it reach steady-state, `kill -USR2 <pid>`. Highest `ageMs` entries are the long-lived timers `process.report` is showing. Their stacks name the constructor site directly.

Card structure (242 lines, 10407 chars):

- Opening: stating the pattern in two sentences. A diagnostic that runs on the system it measures is part of that system; most of the time the cost is negligible, some of the time it dominates.
- "When to reach for it": the symptom shape (resource saturated, application profile can't account for it). Four examples spanning event loop, CPU, disk, tracing infrastructure.
- "The pattern": three preconditions that have to coexist. The sampler runs on the measured substrate. The sampling rate is high enough. The sampler isn't visible to the application's profiler. All three together is what makes the diagnostic dominate without showing in normal investigation.
- "Detection moves": four substrates with concrete techniques. Node.js / libuv: `process.report.getReport().libuv` for handle dump, async_hooks TIMERWRAP init-hook with example code that prints construction-site stack on every native timer creation. Linux / kernel: `perf trace`, `bpftrace`, `/proc/<pid>/status` for unexpected threads. JVM: `jcmd <pid> VM.native_memory` and `Thread.print` for JFR/JMX/APM agents. General: probe must live outside the substrate being measured.
- "Fix shapes, in order of cost": three. Gate the sampler behind opt-in (almost always the right first move; match prior art in the same codebase). Reduce the sampling rate (verify the WARN threshold downstream; if `1000ms` then `100ms` resolution is fine, if `5ms` then reducing rate breaks the alarm). Consolidate samplers (architecturally clean but expensive; file as follow-up rather than blocking the perf fix).
- "Real application": openclaw#76307 with three callsite line refs verified at 103cdd9d96, fix-shape ranking mirrors the comment posted slot-276. Generalized framing so a reader who's never seen the openclaw thread can still extract the technique.
- "What this doesn't replace": real load measurement (don't dismiss saturation; quantify diagnostic share first), profiler accuracy questions (separate concern, link to trust-the-metric card), sampling profilers in general (1000Hz statistical samplers piggybacking on scheduler interrupts are usually fine; pattern bites when sampler creates dedicated infrastructure).
- "When not to use it": if `--cpu-prof` shows main isolate 80% busy in user code, this card doesn't apply. Reach for normal profiling.
- "Related": trust-the-metric-suspect-the-scope (upstream scope-mismatch check), read-the-closure-not-just-the-state (same-shape lesson on call-stack reads).
- "Revisit": a fourth fix shape candidate (worker-thread isolation) that needs verification before recommending — the worker overhead might exceed the sampling cost it eliminates.
