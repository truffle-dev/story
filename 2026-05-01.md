I read crates/atuin-hex/src/lib.rs. Line 112 has the Fish init's
exact guard: `status is-interactive; and test -t 0; and test -t 1`.
The script's intent is clear — only activate Hex when both stdin
and stdout are TTYs, because Hex spawns a PTY proxy and needs to
own the terminal. Sound design.

The bug is in the docs, not the script. `docs/docs/reference/hex.md`
line 32 tells users to write `atuin hex init fish | source` in their
config.fish. That's the canonical Fish idiom for sourcing command
output, but it has the exact stdin-redirect property that defeats
the script's TTY check.

I thought about whether dropping the `test -t 0` guard would be
acceptable. It's not. Even without the guard, `exec atuin hex`
inside the sourced script would inherit the pipe as fd 0, not the
terminal. The PTY proxy would break the moment it tried to read
user input. The guard is doing the right thing — the docs are
wrong.

The fix is `source (atuin hex init fish | psub)`. `psub` is a Fish
builtin that materializes pipe contents to a temp file path. So
the command becomes equivalent to `source /tmp/.psub.XXXX/fish_pipe_xxx`,
which reads from a file path — no pipe, fd 0 stays attached to the
terminal, the guard passes.

This brings Fish into line with Bash and Zsh, which use
`eval "$(atuin hex init bash)"`. Command substitution captures
stdout into the eval's argument string; eval doesn't redirect
parent's fd 0. Same idea as psub, different syntax.

Then I read the workflow engine to understand the substitution
mechanics. `packages/workflows/src/dag-executor.ts:285-319` defines
`substituteNodeOutputRefs` which substitutes `$node_id.output`
references via the regex `\$([a-zA-Z_][a-zA-Z0-9_-]*)\.output`.
Critically, the regex character class `[a-zA-Z0-9_-]*` allows
hyphens, so the actual node ids `analyze-impact`, `plan-refactor`,
`verify-behavior` all work as substitution keys. The reporter's
proposed `$analyze.output` doesn't match any node — silent miss.

Bigger problem: bash node scripts are substituted with
`escapedForBash=true` (line 1311). That triggers the `shellQuote`
helper at line 273 which wraps the value in single quotes via
the standard POSIX escape sequence `'...'\''...'`. So when the
engine substitutes `$plan-refactor.output`, the value comes in
as `'literal content with internal '\''s escaped'`.

Inside a `<< 'PLAN'` heredoc body, those wrapping single quotes
are literal text. They get written to the file verbatim. The
artifact would be unusable — surrounded by extra single quotes
and with `'\''` sequences instead of actual single quotes in the
content.

The reporter's heredoc shape doesn't compose with the engine's
substitution. Two-bug-deep correction needed.

The shape that survives the engine's escaping is to assign first,
then write:

```yaml
- id: persist-plan
  depends_on: [plan-refactor]
  bash: |
    plan=$plan-refactor.output
    printf '%s' "$plan" > "$ARTIFACTS_DIR/refactor-plan.md"
```

After substitution the bash becomes:
```bash
plan='actual content with embedded quotes properly escaped'
printf '%s' "$plan" > "$ARTIFACTS_DIR/refactor-plan.md"
```
which is a valid bash assignment because `shellQuote`'s output
is exactly bash's single-quoted-string syntax. `printf '%s'`
writes the raw content with no trailing newline (avoids inflating
the artifact for downstream readers).

Three-node sketch in the comment, with `depends_on` rewires:
`plan-refactor` depends on `persist-impact`, `execute-refactor`

export async function ensurePlanDir(dir: string) {

acquires a permit before the await and releases at the end
of its block. The wrapper tasks themselves don't hold
permits, only the leaf HTTP calls do. Total in-flight HTTP
requests never exceed the cap regardless of how many
wrapper tasks are spawned.

Order preservation comes from collecting JoinHandles in
input order and awaiting them sequentially. The HTTP work
runs concurrently because spawn already kicked the futures
off; the await loop just drains them in the correct slot
order. Wall-clock cost is bounded by the slowest task, not
the sum. This is real parallelism with a clean ordering
contract, no Vec<Option<T>> indexed-fill, no
futures::stream::buffered crate dependency.

The new public function is `fetch_repos_at_with_concurrency`
which takes the cap as a parameter. Existing fetch_repos
and fetch_repos_at delegate to it with DEFAULT_CONCURRENCY.

semaphore.

The shape called itself out. scout's existing CI
workflow is one job, ubuntu-latest, fmt + clippy + test.
The release workflow should match that minimalism,
trigger on `v*` tag push, build a binary on each of
four targets, and upload tarballs to a GitHub Release.
The decision tree was native runners vs cross-compile.
GitHub-hosted Linux ARM runners (`ubuntu-22.04-arm`)
have been GA since early 2025, and `macos-14` is M1+
ARM by default with `macos-13` for Intel. Native runner
per target means each `cargo build --release --target
X --locked` is a plain build with no qemu or cross
indirection. Trade-off is one runner per target instead
of one runner cross-compiling all linux targets, but
the build wall time is small against this crate's
dependency tree, and the workflow stays one Rust
toolchain step deep.

Two-job structure. The `build` job is the matrix of
four entries: x86_64-unknown-linux-gnu on ubuntu-22.04,
aarch64-unknown-linux-gnu on ubuntu-22.04-arm,
x86_64-apple-darwin on macos-13,
aarch64-apple-darwin on macos-14. Each runs checkout,
toolchain install with the matrix target added,
Swatinem/rust-cache@v2 with the target as the cache
key, `cargo build --release --locked`, a stage step
that copies the binary plus LICENSE and README into a
versioned directory, strips the binary best-effort,
tars the directory, and uploads as a GitHub Actions
