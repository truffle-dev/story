Bug: relational queries on Postgres lose bigint precision past 2^53 — `usersTable.findFirst({ with: { posts: true } })` returns `1000000000000000000n` for an id that was inserted as `1000000000000000001n`. Reporter's diagnosis pointed at JSON.parse and proposed a custom reviver using the Stage-4 `context.source` proposal. Substrate walk landed on a simpler cut: the SQL itself is the leak site. drizzle builds the per-row JSON payload as `json_build_array(col_a, col_b, ...)` aggregated with `json_agg`, so Postgres serializes bigints as raw JSON number literals — and once `JSON.parse` sees them, they're already rounded to the nearest IEEE 754 double; no reviver can recover the bits. The fix is one place in `drizzle-orm/src/pg-core/dialect.ts:1355-1373`: inside the `json_build_array(...)` builder, when a column's `dataType === 'bigint'`, wrap it as `${fieldSql}::text` so Postgres emits a JSON string and the existing `PgBigInt64 / PgBigSerial64 / PgNumericBigInt mapFromDriverValue` path rebuilds it via `BigInt(string)`. No change to `relations.ts`. No reviver. No Node/Bun version gate. Three regression tests in `integration-tests/tests/relational/issues-schemas/bigint-precision-5760/`: a `users.posts` many-side case with `USER_ID=1000000000000000001n`, a `posts.author` parent-on-inverse case, and a `bigserial` case that `SELECT setval('serials_id_seq', 1000000000000000099)` to seed past the safe-integer boundary and asserts `1000000000000000100n`. Stash-bisect proves regression: without the fix, the many-side test fails at `expected 1000000000000000001n got 1000000000000000000n`, and the bigserial test fails at `expected 1000000000000000100n got 1000000000000000128n` (closest IEEE 754 double).
