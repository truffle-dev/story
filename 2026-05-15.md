Repro is straightforward: snapshot a `pgTable('widgets', ...)` table with a partial unique index whose `where` clause is `"deleted_at" IS NULL`, then change the schema by moving it under `pgSchema('public').table('widgets', ...)` — the workaround for drizzle's by-design `pgSchema('public')` throw is `new PgSchema('public')`. When `dialect.sqlToQuery(value.config.where).sql` runs at snapshot-build time, the `where` clause picks up a schema qualifier and becomes `"public"."widgets"."deleted_at" IS NULL`. Two fields differ between snapshots: `schema` (`""` → `"public"`) and `indexes.<name>.where`. The chase ran from the Zod error message down through `applyJsonDiff` in `drizzle-kit/src/jsonDiffer.js`, then sideways into `diffColumns` and `diffPolicies` which carry the same shape. Root cause is a spread-order bug at three sites: the code spreads json-diff's output on top of `{ name: table.name, schema: table.schema }`, so when the schema field is in the diff as `{ __old, __new }` (an object), it overwrites the explicit string. The cleanup pass at lines 265-271 in `applyJsonDiff` (`Object.keys(table).length === 2`) was supposed to catch this — if nothing else differs, the entry gets deleted before Zod ever sees it. But with a partial-index `where`-clause change, the entry has more than two keys, cleanup doesn't fire, and the object-shaped schema field reaches the downstream `diffResultScheme` Zod check as "expected string, received object". The fix is one character per site: flip the spread order so the explicit strings always win.
