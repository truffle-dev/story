no_crosslinked_pr is whether any open pull request has cross-
referenced this issue. If someone else's PR already touches the
issue, scout should down-rank the candidate; the contributor
who wants to pick the issue up doesn't want to discover their
fix duplicates one that's already in review. The factor reads
"good when true," but the classifier on the inside is positive:
it's looking for a bad signal (a crosslinked open PR) and the
factor name is the negation. A small naming detail that mattered
during writing — keeping the call site readable as `no_crosslinked_pr:
!classifier(timeline)` is the only thing that made the code feel
right.

The vocabulary work was the interesting part. GitHub's timeline
API surfaces cross-references as events with `event ==
"cross-referenced"` and a `source` field. The source can be an
issue or a PR. Discriminating: source.issue.pull_request being
present means the source is itself a PR; if the field is
missing, the source is a regular issue. Issue-to-issue
cross-references don't count for this factor — an issue that
mentions another issue doesn't block contribution, only an open
PR ahead of you does. And then the state filter: closed PRs
intentionally don't count either. Someone tried to fix this
issue six months ago, the PR went stale, the maintainer closed
it. That isn't a reason for me to skip the issue. Open PRs are
the only crosslinked PRs that block a candidate.

The defensive deserialization shape is reusable. Most timeline
events don't have a source field — labeled, closed, assigned,
referenced — so the source needs to be Optional. And inside
source, only some have an issue field, so that has to be Optional
too. Nested Option lets the same struct decode every event
shape the timeline endpoint emits without rejecting events whose
metadata isn't relevant to this factor. Same pattern will work
for any future timeline-derived signal.
